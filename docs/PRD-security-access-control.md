# PRD: Admin Security & Access Control

## 1. Summary
Covers authentication, authorization, and audit logging for **human admins/operators** using the panel UI — distinct from the bot/reseller API-key auth already specified in [PRD-telegram-bot-api.md](PRD-telegram-bot-api.md) §5. Both systems feed the same audit log.

## 2. Goals
- Least-privilege access for support staff who don't need full control.
- A complete, tamper-evident record of who changed what.
- Protect the single highest-value target in the system: super-admin credentials and node join-token issuance.

## 3. Non-Goals
- Bot/reseller API-key authentication and scoping — covered in [PRD-telegram-bot-api.md](PRD-telegram-bot-api.md) §5; this PRD is human-admin login only.
- The screens themselves (login form, user management table) — covered in [PRD-admin-panel-ux.md](PRD-admin-panel-ux.md) §4.6; this PRD defines the enforcement rules those screens must respect.

## 4. Actors & Roles

| Role | Permissions |
|---|---|
| **Super Admin** | Full access: accounts, nodes, API keys, admin user management, role assignment, audit log |
| **Operator** | Full access to accounts and nodes; cannot manage API keys, admin users, or roles |
| **Support (read-only)** | View accounts/nodes/usage; cannot create/update/delete/suspend anything |

Roles are enforced server-side on every endpoint (not just hidden in the UI).

## 5. Super Admin Bootstrap & CLI Administration

The panel ships with no default credentials baked into any image — a Super Admin is only created through an explicit action, either during installation or later via the CLI.

- The API exposes an `/internal/*` route group used exclusively by the `wgpanel` CLI running on the panel host. This includes `POST /internal/admins` (create a Super Admin) and `GET /internal/healthz` (readiness, richer than the public `/healthz` below).
- Trust boundary, two independent layers — **not just one**:
  1. **Not reachable from the public internet**: the port is published as `127.0.0.1:${API_PORT}` in `deploy/docker-compose.yml`, and Caddy's config (`deploy/Caddyfile`) never proxies `/internal/*` at all.
  2. **A shared secret (`INTERNAL_API_TOKEN`), checked on every `/internal/*` request** via an `X-Internal-Token` header, compared in constant time. This layer exists because layer 1 alone is not sufficient: Docker's default bridge network lets any *sibling* container in the same compose project (`frontend`, `caddy`) reach `api:8080` directly over the internal network regardless of what's published to the host or what Caddy proxies — a compromised frontend container could otherwise call `http://api:8080/internal/admins` itself. The token is generated by `install.sh` alongside the other secrets, stored in `.env` (never in an image), and read from there by both the API container and the `wgpanel` CLI.
  - The overall trust basis is still "whoever can read `.env` on the panel host is already as privileged as a DB root login or `docker exec`" — the token doesn't add a *new* secret to protect, it closes the specific gap that host-only network exposure leaves open between containers.
- Separately, the API also exposes `GET /healthz` — **public, unauthenticated, proxied by Caddy** — returning a minimal `{"status":"ok"}` once the process is up and DB/Redis are reachable. This is *not* a security control; it's the target for Docker's own container healthcheck (`deploy/docker-compose.yml`), external uptime monitors, and any future load balancer, and intentionally reveals nothing about migrations, admin bootstrap state, or version. `/internal/healthz` is the richer, loopback-only variant used by `wgpanel` and the installer for readiness gating.
- **Installation flow**: `deploy/install.sh` prompts for a desired admin username, brings the stack up, polls `/internal/healthz` until the API and its migrations are ready, then calls `POST /internal/admins` with that username and a freshly generated random password. The resulting credentials are printed once, in a clearly delimited block, at the very end of the install script's output — not before the account is confirmed to exist.
- **Post-install**: `wgpanel create-admin [--username NAME]` calls the same endpoint at any time (e.g., to add a second Super Admin, or after losing access to the first one). It always creates a *new* admin row rather than resetting an existing one's password — password resets for an existing admin are a distinct, not-yet-specified flow.

### Acceptance Criteria
- Given a fresh installation with an empty admin table, running `install.sh` end-to-end results in exactly one Super Admin existing in the database, and the CLI output the operator sees contains that exact username and password (never a placeholder, never "check the .env file").
- Given `/internal/admins` is called, the response (and the install script's terminal output sourced from it) never appears in any persisted log file at `info` level or above — only in the interactive terminal.
- Given `deploy/docker-compose.yml`'s Caddyfile config, a request to `https://<panel-domain>/internal/anything` from the public internet returns 404 (route doesn't exist at that listener) — this layer is unreachable by construction, not by an auth check that could be misconfigured.
- Given a request to `/internal/*` from *within* the Docker network (e.g., simulating a compromised `frontend` container calling `api:8080` directly) without a valid `X-Internal-Token`, the API returns `401 Unauthorized` — the network-topology layer above is not the only thing standing between a sibling container and admin creation.
- Given `wgpanel create-admin` is run twice with different usernames, both Super Admins exist independently and either can log in.
- Given a request to the public `GET /healthz`, the response contains no information beyond process/DB/Redis liveness — no version string, no migration state, no admin count — even though `/internal/healthz` may expose all of that on the loopback-only path.

## 6. Authentication

- Username/password (bcrypt/argon2 hashed) + optional TOTP-based 2FA per admin account.
- Session model: short-lived JWT access token (15 min) + rotating refresh token (7 days), refresh tokens revocable individually (stored server-side in Redis with a revocation flag) so a compromised session can be killed without forcing a password reset.
- Optional IP allowlist for admin-panel login, configurable by a Super Admin (off by default, since admins may travel).

### Acceptance Criteria
- Given 2FA is enabled for an account, login requires a valid TOTP code; a stolen password alone is insufficient.
- Given an admin's refresh token is revoked (e.g., by a Super Admin after an offboarding), any further use of that token — and the access tokens minted from it — is rejected within one Redis lookup, no propagation delay.
- Given 5 consecutive failed login attempts for an account within 10 minutes, the account is temporarily locked for 15 minutes and a security alert is logged.
- Given the IP allowlist is enabled and a login attempt comes from outside it, the request is rejected with `403` before password verification (no timing signal on whether the password was correct).

## 7. Authorization

- Every API route declares its required role(s); a Support-role token calling a mutating endpoint receives `403 Forbidden`.
- Node join-token generation and API-key issuance are Super-Admin-only actions regardless of any future role additions — these are the two actions that mint new trust into the system.

### Acceptance Criteria
- Given an Operator calls "create API key" or "add admin user," the request is rejected with `403` even though Operators can otherwise manage accounts/nodes.
- Given a role is changed for a logged-in admin, the change takes effect on their **next** request (access tokens are short-lived enough that this isn't a meaningful window) — no requirement to force-invalidate their current token, but a Super Admin can force-revoke immediately via §6.

## 8. Audit Log

- Every state-changing action — account create/update/suspend/delete, node add/remove/edit, API key issue/rotate/revoke, admin user/role changes, login/logout, failed logins — is written to an append-only audit log: `actor` (admin id or API key id), `action`, `target`, `before/after diff` where applicable, `ip_address`, `timestamp`.
- Audit log is queryable in the admin UI, filterable by actor/action/date range/target, and itself read-only (no delete/edit endpoint exists for it — retention/pruning, if ever needed, is an infra-level job, not an application feature).

### Acceptance Criteria
- Given any mutating action from either the UI or the bot API, a corresponding audit entry exists within 5 seconds, queryable by the acting identity.
- Given an audit log query for a specific account, all lifecycle events for it (from creation through any suspends/renews/deletes) are returned in chronological order regardless of whether they originated from an admin or a bot API key.

## 9. Non-Functional Requirements
- Passwords hashed with argon2id (or bcrypt cost ≥ 12 as a fallback); never logged in plaintext anywhere, including audit entries.
- All admin-panel traffic served over TLS only (enforced by the Caddy layer in `deploy/`).
- Audit log storage is append-only at the database level (no `UPDATE`/`DELETE` grants for the application's DB role on that table).
