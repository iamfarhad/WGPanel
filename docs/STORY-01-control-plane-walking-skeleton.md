# Story 1: Control-Plane API Walking Skeleton

## Story

**As** an operator running `deploy/install.sh` on a fresh server,
**I want** the control-plane API to actually exist and answer the endpoints the installer and `wgpanel` CLI already call (`/healthz`, `/internal/healthz`, `/internal/admins`, admin login),
**so that** a fresh install produces a real, working panel with a verified super admin — not `docker compose up` pointed at images that don't exist yet.

## Why this one first

Every other roadmap item (node agent, bot-facing API, frontend) depends on the control-plane API existing at all. More specifically, `deploy/install.sh`, `deploy/wgpanel`, and `deploy/docker-compose.yml` already **hard-code a contract** — `GET /healthz`, `GET /internal/healthz`, `POST /internal/admins`, an argon2-hashed `admins` table, an append-only `audit_log` table — none of which exists yet. This story is the smallest slice that makes the deployment tooling already written actually true, rather than adding a fourth thing (frontend, node agent, bot API) on top of an unverified foundation.

Everything in scope here is already specified — nothing new is invented — in:
- [PRD-security-access-control.md](PRD-security-access-control.md) §4–§9 (roles, bootstrap, authentication, authorization, audit log)
- [PRD-account-management.md](PRD-account-management.md) §4 (data model conventions this schema style should follow)

## Definition of Done

- [x] `docker compose up -d` against a real built `wgpanel-api` image brings `db`, `redis`, and `api` to `healthy` status. Verified locally: `db`/`redis`/`api` all report healthy, `api` boots and runs its migrations cleanly against a real TimescaleDB container.
- [x] `curl http://<host>/healthz` → `200 {"status":"ok"}`, unauthenticated, no version/migration/admin-count leakage. Verified.
- [x] `curl http://127.0.0.1:$API_PORT/internal/healthz` → readiness + migration status; requires `X-Internal-Token`. Verified, including the scenario that mattered most: a request from *another container on the same Docker network* (simulating a compromised `frontend`/`caddy`) hitting `api:8080/internal/admins` directly, with no token, was confirmed rejected with `401` — this is the specific gap the loopback-only design would have missed (see §5 of the Security PRD and the design note below).
- [x] `POST /internal/admins` creates a row, hashes the password (argon2id, confirmed never stored or logged in plaintext — grepped container logs for the test password, zero matches), writes an `audit_log` row, and rejects duplicate usernames with `409`.
- [x] `POST /api/v1/auth/login` issues a JWT + Redis-backed refresh token for correct credentials, and returns the identical `401 invalid_credentials` shape for both "wrong password" and "unknown username" (no enumeration signal). Verified end-to-end against a real Postgres + Redis.
- [x] Every action above writes a corresponding `audit_log` row. Verified (`admin.created`, `admin.login`).
- [x] A CI workflow (`.github/workflows/build-api.yml`) builds/tests/publishes the image to the registry path referenced in `deploy/.env.example` (`API_IMAGE`) — written, not yet run for real since it needs to live on a pushed branch.
- [ ] The same pass against the real `https://<domain>/healthz` path through Caddy's TLS, and a full `install.sh` run as root — needs a real domain + a disposable VM, neither of which exists in this environment. Manual follow-up (Task 17 below); the underlying HTTP contract it depends on is already verified above.

### Design correction made during implementation

The original plan for `/internal/*` protection was "loopback-only bind + Caddy doesn't proxy the path." While implementing the middleware, that turned out to be an incomplete boundary: Docker's bridge network lets any sibling container in the same compose project (`frontend`, `caddy`) reach `api:8080` directly over the internal network, regardless of what's published to the host or what Caddy routes — a compromised `frontend` container could otherwise call `POST /internal/admins` itself. Added a second, independent layer: a shared-secret `INTERNAL_API_TOKEN` (generated by `install.sh`, checked in constant time by `internalOnly` middleware) that the network-topology argument doesn't depend on. `docs/PRD-security-access-control.md` §5 was updated to describe both layers accurately instead of overselling the first one alone. This was verified directly: a container attached to the compose network calling `api:8080/internal/admins` without the token gets `401`.

## Tasks

### A. Project scaffolding
1. Initialize the Go module under `backend/` (`cmd/api`, `internal/{config,db,httpapi,auth,audit}`).
2. Config loader reading exactly the env vars `deploy/docker-compose.yml` already sets: `POSTGRES_DSN`, `REDIS_URL`, `JWT_SECRET`, `API_HMAC_MASTER_KEY`, `NODE_AGENT_GRPC_PORT`, `NODE_AGENT_JOIN_TOKEN_TTL_MIN`.
3. Postgres connection pool + a migration runner (e.g. `golang-migrate`); first migration enables the `timescaledb` extension (needed later, harmless now).
4. Redis client setup.

### B. Schema
5. Migration: `admins` (id, username unique, password_hash, role, totp_secret nullable, created_at).
6. Migration: `audit_log` (append-only — DB role for the app should not have `UPDATE`/`DELETE` grants on this table, per Security PRD §9).

### C. HTTP surface
7. Router with two route groups: public (what Caddy proxies) and `/internal/*`. Add IP-allowlist middleware on the `/internal/*` group that rejects anything not from loopback as defense-in-depth — Caddy's config already makes this unreachable externally by not proxying the path at all, but the app shouldn't rely solely on the reverse proxy's config for this guarantee.
8. `GET /healthz` (public): DB ping + Redis ping, minimal `{"status":"ok"}`, no auth.
9. `GET /internal/healthz`: same liveness checks plus `migrations_applied` and `admin_bootstrapped`.
10. `POST /internal/admins`: create a super admin — hash password with argon2id, insert, write an audit_log row, return the created username (never echo or log the password server-side).
11. `POST /api/v1/auth/login`: verify credentials, issue a 15-minute JWT + a rotating refresh token stored in Redis with a revocation flag (Security PRD §6).
12. Structured logging with secret redaction — passwords, JWT secret, HMAC key, and DB/Redis passwords must never reach a log line (verify by grepping log output in task 15).

### D. Containerization & CI
13. Multi-stage Dockerfile for `backend/`. Prefer adding a `healthcheck` subcommand to the same binary (`wgpanel-api healthcheck`) over depending on `wget`/`curl` being present in the final image — cleaner than the `wget`-based probe currently in `deploy/docker-compose.yml`, which was written as a placeholder pending this decision.
14. Once the healthcheck subcommand exists, update `deploy/docker-compose.yml`'s `api.healthcheck.test` to use it instead of `wget`.
15. GitHub Actions workflow: build and push to the registry path in `deploy/.env.example`'s `API_IMAGE` on merge to main / tag.

### E. Verification
16. Local smoke test: build the image, run `docker compose up -d` with a `build:` override pointed at `backend/`, then run `wgpanel doctor` and confirm every check is green.
17. Full end-to-end pass of `deploy/install.sh` against the real image on a throwaway VM, confirming the printed super-admin credentials actually work against `/api/v1/auth/login`.

- 2FA/TOTP, IP allowlisting for admin login, password-reset flow — deferred pieces already flagged as such inside the Security PRD.
